# 04. 查询执行与工具系统

## 1. `query()` 是代理内核

`src/query.ts` 是整个系统的执行循环核心。

它负责一个完整 agent turn：

1. 接收消息与上下文
2. 构造模型请求
3. 流式消费模型输出
4. 识别 `tool_use`
5. 执行工具
6. 回灌 `tool_result`
7. 判断是否继续下一轮
8. 在需要时处理 compact、stop hooks、retry、fallback

但当前实现已经不止这条最短主线。
在工具执行之后，`query()` 还会继续处理：

- 当前 turn 内可消费的 queued commands
- `task-notification` 附件注入
- 基于 `agentId` 的主线程 / 子代理队列隔离
- memory prefetch 注入
- skill discovery prefetch 注入

## 2. Query 依赖注入

`src/query/deps.ts` 将部分关键 I/O 能力抽成 `QueryDeps`：

- `callModel`
- `microcompact`
- `autocompact`
- `uuid`

当前 scope 是“克制型依赖注入”：

- 已足够支持核心测试替换
- 但没有把整条 query 链路完全参数化

## 3. 查询循环主数据流

```text
messages + context
  -> build system prompt / userContext / systemContext
  -> deps.callModel()
  -> 流式接收 assistant 输出
  -> 收集 tool_use block
  -> 执行工具
  -> 产生 tool_result
  -> 回到 query 下一轮
  -> stop hooks / compaction / retry / fallback
  -> 输出最终消息流
```

几个实现细节需要单独记住：

- 代码不会只依赖 `stop_reason === 'tool_use'`
- 流式阶段会主动收集 `tool_use` block
- query 会按配置决定是否启用 `StreamingToolExecutor`
- 工具执行后还会尝试把当前可消费的 queued commands 转成 attachments
- 主线程只消费 `agentId===undefined` 的队列项，子代理只消费发给自己的 `task-notification`

## 4. 模型调用层

关键实现位于：

- `src/services/api/claude.ts`
- `src/services/api/client.ts`

主链路可概括为：

```text
query()
  -> queryModelWithStreaming()
  -> getAnthropicClient()
  -> 选择 provider 客户端
  -> 规范化为内部消息 / 流事件
```

当前代码里明确支持的 provider 适配包括：

- Anthropic
- Bedrock
- Vertex
- Foundry

## 5. 工具契约

`src/Tool.ts` 定义统一工具契约：

- `Tool`
- `ToolUseContext`
- `ToolResult`
- `Tools`
- `buildTool()`

这层统一了：

- 输入 schema / JSON schema
- prompt 描述
- 调用入口
- 并发安全声明
- 只读 / destructive / permission 检查
- deferred tool 与 ToolSearch 相关元信息

## 6. `ToolUseContext`

`ToolUseContext` 是工具运行时最关键的上下文对象。

当前可确认的重要内容包括：

- `options.commands`
- `options.tools`
- `options.mcpClients`
- `options.mcpResources`
- `options.refreshTools`
- `getAppState()` / `setAppState()`
- `abortController`
- 文件状态缓存
- 通知 / UI / elicitation / attribution / history 更新接口
- 当前消息列表

这里不是单纯“工具参数”，而是模型工具、权限、MCP、AppState 与 UI 的汇合点。

## 7. 工具注册中心

`src/tools.ts` 负责按环境、权限和特性开关组装最终工具池。

除了传统文件与 shell 工具，当前代码里已经明确进入工具池的重要能力还包括：

- `AgentTool`
- `TaskOutputTool`
- `TaskStopTool`
- `TodoWriteTool`
- `AskUserQuestionTool`
- `EnterPlanModeTool`
- `ExitPlanModeV2Tool`
- `EnterWorktreeTool`
- `ExitWorktreeTool`
- `TaskCreateTool` / `TaskGetTool` / `TaskUpdateTool` / `TaskListTool`
- `SendMessageTool`
- `TeamCreateTool` / `TeamDeleteTool`
- `ToolSearchTool`
- `ListMcpResourcesTool` / `ReadMcpResourceTool`

在特性开关开启时，工具池还会继续扩展到：

- `WorkflowTool`
- cron tools
- `RemoteTriggerTool`
- `MonitorTool`
- `WebBrowserTool`
- `REPLTool`

因此“工具系统”不只是 Bash、Read、Edit、WebFetch，而是已经覆盖：

- 文件操作
- shell 执行
- 计划模式
- worktree
- 任务与协作代理
- task / todo 生命周期
- workflow / trigger / monitor
- MCP 资源访问
- deferred tool 检索

## 8. 工具执行编排

`src/services/tools/toolOrchestration.ts` 负责工具批次编排。

核心规则：

- `isConcurrencySafe=true` 的连续片段可并发执行
- 其他工具按批次串行执行
- 并发批次的 `contextModifier` 会延后统一合并

并发上限受：

- `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY`

控制。

## 9. 流式工具执行

`src/services/tools/StreamingToolExecutor.ts` 提供另一条工具执行路径。

它的作用不是替代工具编排，而是：

- 在流式响应尚未结束时尽早启动可执行工具
- 维护执行中工具状态
- 在允许的情况下提前推进工具结果与上下文更新

## 10. Deferred tools 与 ToolSearch

当前工具系统还存在“延迟暴露”的一层：

- 工具可以声明 `shouldDefer`
- 工具也可以声明 `alwaysLoad`
- `ToolSearchTool` 负责把 deferred tools 暴露给模型检索与选择

这意味着“模型看到的工具集”不一定等于“最终可用工具全集”。
